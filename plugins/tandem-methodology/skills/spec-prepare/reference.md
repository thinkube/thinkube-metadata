## AC verifiability тАФ anti-patterns & reframes

**Every AC must be AI-verifiable _at the gate it arms_** тАФ i.e. the verifier can check it _before_ the step that gate guards (Done before merge; acceptance before merge). The human's only gate is acceptance (judging the assembled result), never executing a check. As each AC lands, ask: _"what actor and environment does verifying this need, and is that available before the gate this AC arms?"_ Reject/reframe two anti-patterns:

- **Human-executed** тАФ "manual verify in a fresh session," "the human checks," "you confirm thatтАж". Reframe: for a probabilistic/heuristic outcome, gate its **controllable proxy** plus an **AI-run probe** (e.g. for LLM routing: "the description carries the proven trigger shape" + an AI discrimination pass over positive/negative cases). If it genuinely can't be AI-verified, it's a **Design/Constraints note**, not an AC.
- **Deploy/merge-circular** тАФ "after deployingтАж," "once merged, the live/production endpointтАж" тАФ verifiable only by performing the merge/deploy the AC is supposed to authorize (the loop is verify тЖТ Done тЖТ accept тЖТ merge тЖТ deploy, so this deadlocks). Reframe into (a) a **pre-merge-verifiable** AC checked in a local run / ephemeral PR-preview / staging environment, **plus** (b) a **post-deploy smoke check** recorded as a follow-up obligation or a follow-up slice тАФ _never_ a Done-blocking AC. Do not reorder the loop to deploy before acceptance.
- Example тАФ Bad (human-executed): "In a fresh session, confirm the new prompt routes correctly." тЖТ Good (AI-verifiable): "Given the new prompt, a routing-discrimination pass routes 5/5 positive utterances to the skill and 0/4 negatives тАФ checked by the verifier." Bad (deploy-circular): "After deploying, GET /health returns 200." тЖТ Good: "GET /health returns 200 against the PR-preview/staging deploy (Done-gating); production /health 200 is a post-deploy smoke check, not a Done condition."

## The AC-verifiability auditor (Procedure step 7, in full)

Once the ACs and Design are settled, run an **AC-verifiability auditor** тАФ the **opening** AI-verification gate (TEP-tgzx3p), the authoring-time mirror of the closing gate SP-tgzyfy shipped. A Spec reaches **Ready only when every AC is certified `verifiable` and carries a runnable `ac_verifications` entry.** **Reuse the methodology's adversarial-subagent pattern** тАФ the same delegation shape as `reviewer` / `verifier`: hand the drafted `## Acceptance Criteria` (plus the Design and the step-5 verification recipe) to an adversarial pass via the `Task` tool, so each AC is interrogated with **independent context**, not waved through by the author who just wrote it. For **each** AC the auditor asks the three questions:

- **Actor** тАФ who runs the check? The **AI verifier** (a shell / playbook probe), or тАФ for a prose AC тАФ SP-7's independent **assessor**. Never a human.
- **Environment** тАФ where does it run: `local` (the repo / a shell), `cluster` (an infra lifecycle), or `assessment` (SP-7's assessor reads the artifact and grades it)?
- **Availability before the gate it arms** тАФ is that actor + environment available _before_ the step this AC guards (Done before merge; acceptance before merge)?

The auditor returns, per AC, one of **three verdicts** тАФ `verifiable`, `assessment`, or `needs-reframe(why)`:

- **`needs-reframe`** тАФ the AC is **human-executed** or **deploy/merge-circular** (the step-4 anti-patterns), so no AI actor + environment can check it before its gate. **Rework or drop it before emitting the map** тАФ reframe per step 4 (probabilistic тЖТ controllable proxy + AI probe; deploy-circular тЖТ pre-merge/preview AC + a non-gating post-deploy smoke check), land the new wording with a `get_thinkube_file` тЖТ `write_spec` read-modify-write, then **re-audit**. A `needs-reframe` AC **never gets a verification entry** тАФ so a partial map can't sneak it past the gate.
- **`verifiable`** тАФ a **runnable** AC. Produce a concrete **`{ run, env }`**: `run` is the shell / playbook command that checks _this_ AC (exit 0 = pass), **pointing at the slice's held-out `acceptance/` probe** (`acceptance/SP-{n}_SL-{m}.*`, the path SP-7's `ACCEPTANCE_EVIDENCE_RE` fence keeps independent of the code-author). SP-7's auditor now **keeps that acceptance command as-is** rather than overriding it to the repo's `npm test` default тАФ the probe _is_ the check. `env` is `local` (repo / shell) or `cluster` (an infra lifecycle), grounded in the repo's verification recipe (the codebase facts gathered in step 5).
- **`assessment`** тАФ a **prose / UX / skill** AC that no runnable probe fits: the outcome is a judgment about an artifact (an instruction reads clearly, a methodology behaves as described, a doc conveys the intended shape) rather than an exit code. Produce **`{ env: "assessment" }`** тАФ no `run`. This is SP-7's new **`assessment` AC kind**: at the closing gate an **independent assessor** reads the relevant artifact and grades the criterion, standing in for a shell probe. Use `assessment` only when the outcome genuinely can't be reduced to a controllable proxy + AI probe (which would be `verifiable`); it is not an escape hatch for a `needs-reframe` AC.

When тАФ and only when тАФ **every** AC is `verifiable` **or** `assessment`, emit the map:

```
write_spec { thinking_space: <id>, spec: {n}, ac_verifications: { "1": { run, env }, "2": { env: "assessment" }, тАж } }
```

keyed by **1-based AC ordinal**, **exactly one entry per AC**, every ordinal `1..N` present (no orphan / missing keys). `write_spec` normalizes and serializes it to the Spec's `ac_verifications:` frontmatter; this is the per-AC declaration the **closing** gate (SP-tgzyfy / TEP-tgzx3p) resolves at Spec quiescence and gates Done / commit on all-green тАФ **running** each `verifiable` entry's `run` command and routing each `assessment` entry to SP-7's independent assessor. **Emitting the full map is what arms тЖТ Ready.** A Spec with any un-audited, `needs-reframe`, or undeclared AC **cannot reach Ready** тАФ the тЖТ Ready gate (`createSlice`'s `readyGate`) blocks and names the offending ordinal, so don't skip this step or emit a partial map. **Re-run this whole step whenever the ACs change** (and `/slice` re-checks it when it touches ACs).

## Constraints (full)

- The four section headers (`## Acceptance Criteria`, `## Constraints`, `## Design`, `## File Structure Plan`) are **load-bearing** тАФ the quality gates and the staleness hash look for these exact strings. Don't rename them.
- **Acceptance criteria are outcome-level, not implementation steps.** Each `- [ ]` line is something the user can observe or a verifier can check. Implementation work lives in slices, not here.
- **Acceptance criteria must be AI-verifiable at the gate they arm.** No human-executed ACs ("the human checks in a fresh session") and no deploy/merge-circular ACs (verifiable only after the merge/deploy the AC gates) тАФ both stall the loop. Reframe per step 4: probabilistic тЖТ controllable proxy + AI probe; deploy-circular тЖТ pre-merge/preview-verifiable AC + a non-gating post-deploy smoke check; otherwise тЖТ a Design/Constraints note. The human's only gate is acceptance.
- **Every AC carries a certified `ac_verifications` entry before Ready** (TEP-tgzx3p / closes the SP-tgqf1v gap). The verifiability auditor (Procedure step 7) certifies each AC via an independent adversarial subagent pass and emits, per ordinal through `write_spec`, either a runnable `{ run, env }` (`verifiable` тАФ `run` a held-out `acceptance/` probe, not the `npm test` default) or `{ env: "assessment" }` (`assessment` тАФ a prose AC SP-7's assessor grades); a `needs-reframe` AC is reworked or dropped, **never declared**. The тЖТ Ready gate (`createSlice`'s `readyGate`) blocks any Spec with an un-audited or undeclared AC and names the ordinal тАФ don't hand-craft the frontmatter and don't emit a partial map.
- **One serialization, both ends.** Emit the map only through `write_spec` (which normalizes each entry to its canonical frontmatter тАФ `{ run, env }` for a probe, `{ env: "assessment" }` for an assessment AC); the closing gate reads the same frontmatter. Don't invent a second declaration shape тАФ the round-trip holds by construction.
- Don't invent acceptance criteria the user didn't agree to. Each `- [ ]` line should trace to something the user explicitly said or confirmed.

## Safety / fallback (full)

- **No acceptance criteria the user will commit to.** Refuse to write тАФ at least one user-observable criterion is required, or the тЖТ Ready gate will block the Spec's slices from advancing.
- **An AC the auditor can't certify.** If an AC stays `needs-reframe` after a rework attempt тАФ no AI actor + environment can verify it before its gate тАФ it is **not an AC**: move its intent to a Design/Constraints note or a non-gating follow-up obligation, then drop the `- [ ]` line. Never emit an `ac_verifications` entry for it, and never leave it undeclared "for now" тАФ the тЖТ Ready gate will block the whole Spec on it.
- **Existing spec with user edits.** Re-fetch with `get_thinkube_file` first; preserve sections the user has filled out. `write_spec` rewrites the whole body, so only re-emit sections the user has agreed to update during this run тАФ keep the rest verbatim. Re-run the step-7 audit when the ACs changed.
- **`write_spec` / `get_thinkube_file` absent in this session.** STOP and say so тАФ do **not** fall back to a raw `Write`/`Edit`, which would write the Spec outside the thinking space (into the code repo). Fix: start a fresh session in the repo so `.mcp.json` loads the kanban server.
